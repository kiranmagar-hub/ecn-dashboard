Purpose of this Document: This document outlines the technical steps required to grant SQL Server access to a SharePoint-synced network folder. Please review and advise on the most suitable approach for our environment.

1. Background & Business Need

Our team needs SQL Server to read from and/or write to files stored in a SharePoint synced folder (via Microsoft OneDrive client). Currently, SQL Server's service account does not have permission to access these folders, resulting in access denied errors when SQL jobs or queries attempt to reference these paths.

What is a SharePoint Sync Folder? When a user syncs a SharePoint library via OneDrive, it creates a local folder (e.g., C:\Users\[username]\OneDrive - [Company]\[Library]). SQL Server cannot automatically access these paths because it runs under a service account, not the user's session.

Connect SQL Server directly to SharePoint Online via Microsoft Graph API using an Azure App Registration. This bypasses the sync client entirely and is the most reliable for production workloads.

4. Step-by-Step: Option A (Recommended Starting Point)

Please perform the following steps on the SQL Server host machine:

Step 1 — Identify SQL Server Service Account

  1. Open Windows Services Press Win + R → type services.msc → press Enter
  2. Locate the SQL Server service Find SQL Server (MSSQLSERVER) or the named instance (e.g., SQL Server (SQLEXPRESS))
  3. Note the "Log On As" account Right-click the service → Properties → Log On tab. This is the account that needs folder access.
    Common values: NT SERVICE\MSSQLSERVER, DOMAIN\sqladmin, or Network Service

Step 2 — Grant Folder Permission

  1. Navigate to the SharePoint synced folder e.g., C:\Users\[username]\OneDrive - [Company]\[LibraryName]
  2. Open folder Properties Right-click the folder → Properties → Security tab → Edit
  3. Add the SQL Server service account Click Add → type the service account name → Click Check Names → OK
  4. Set permissions Grant Read (for reading files) or Modify (for read/write). Click Apply → OK.

Step 3 — Verify Access via SQL

Run the following in SQL Server Management Studio (SSMS) to confirm access:

-- Enable xp_cmdshell (if not already enabled) -- DBA action required
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1;
RECONFIGURE;

-- Test folder visibility
EXEC xp_cmdshell 'dir "C:\Users\[username]\OneDrive - [Company]\[FolderName]"';

-- Test file read (example: BCP import)
BULK INSERT dbo.YourTable
FROM 'C:\Users\[username]\OneDrive - [Company]\[FolderName]\data.csv'
WITH (FIELDTERMINATOR = ',', ROWTERMINATOR = '\n', FIRSTROW = 2);

5. Step-by-Step: Option B (Network Share via UNC Path)

  1. Share the synced folder Right-click folder → Properties → Sharing tab → Advanced Sharing → Check "Share this folder" → set a Share Name
  2. Set Share Permissions Click Permissions → Add the SQL Server service account → grant Read or Full Control
  3. Also set NTFS (Security) Permissions Security tab → Edit → Add service account → grant appropriate permissions
  4. Test UNC path from SQL Server EXEC xp_cmdshell 'dir "\\MachineName\ShareName\FolderName"';

8. Security Considerations

Important Notes for DBA Review:
  • Grant minimum necessary permissions (Read-only if write access is not required)
  • xp_cmdshell should only be enabled if no alternative exists — it is a potential security risk
  • Service accounts should not have broader permissions than needed
  • Network shares should be protected by both Share and NTFS permissions
  • Graph API App Registrations should use scoped SharePoint permissions, not tenant-wide
